C++ · Algorithms

TSP Visualization

View on GitHub

Problem

The Traveling Salesman Problem asks for the shortest route visiting all cities exactly once and returning to the start. It's NP-hard — no polynomial-time algorithm is known for all instances. The Christofides algorithm produces solutions within 1.5× the optimum for metric instances.

Implementation

My modified approach builds a minimum spanning tree (MST), then performs a depth-first search recording visit order. This reduces time complexity from O(n³) to O(n² log n) compared to the full Christofides algorithm.

Rust · Godot · Networking

Multiplayer FPS

View on GitHub

About

A real-time multiplayer first-person shooter built as a networking course final project. Players connect from a native desktop client or a browser and fight in a shared 3D arena. All game logic runs on an authoritative Rust server; clients send input and receive world-state updates at ~60 Hz.

Stack

Server: Rust with Tokio async runtime — handles WebTransport (QUIC/HTTP3) and legacy TCP+UDP simultaneously on the same player state.
Client: Godot 4 (GDScript), exported to both native desktop and browser (HTML5). Browser clients use a hand-written JavaScript WebTransport bridge called via JavaScriptBridge.eval().
Protocol: Shared Rust crate with nine hand-serialized binary message types (big-endian, no external serialization library). Deployed on AWS EC2 with Let's Encrypt TLS.

Key Engineering

Dual-transport architecture — the same server accepts WebTransport (browser) and TCP+UDP (desktop) clients in the same session. The transport layer is abstracted so game logic is transport-agnostic.
Authoritative hit detection — the server validates melee hits by computing 3D Euclidean distance; clients cannot self-report damage.
Position reconciliation — the client tracks per-frame drift between its locally-predicted position and the server-authoritative position, exportable as CSV and viewable as a live in-game overlay (F3).
Binary protocol — PlayerInput is 20 bytes; WorldState is 6 + 26 bytes per player. All nine message types implement serialize() / deserialize() with round-trip tests.

Python · Django · AWS

CloudCost

View on GitHub

About

CloudCost tracks and analyzes AWS spend across multiple accounts from a single dashboard. Connect any number of AWS accounts, let Celery Beat poll Cost Explorer on a daily schedule, then explore breakdowns by service, region, and date range — from the web UI, the REST API, or the cloudcost CLI.

Stack

Backend: Django + Django REST Framework with JWT authentication. PostgreSQL stores accounts and cost data; Redis backs the Celery task queue.
Workers: Celery Beat triggers daily polls of the AWS Cost Explorer API; credentials are encrypted at rest with Fernet.
CLI: cloudcost command installed via uv tool — supports init, login, accounts add/list, and costs summary with custom date ranges.
Deployment: Docker Compose bundles the web server, worker, beat scheduler, Postgres, and Redis into a single docker compose up.

Key Features

Multi-account support — add as many AWS accounts as needed; each uses a scoped IAM user with read-only Cost Explorer permissions.
Encrypted credential storage — access keys are Fernet-encrypted before being written to the database.
Flexible querying — filter cost summaries by account, service, region, or arbitrary date ranges via the API or CLI.
Scheduled polling — Celery Beat runs a daily job at 02:00 UTC; one-off polls are available via the CLI or a Django management command.
